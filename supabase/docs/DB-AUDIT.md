## 1. Executive Summary

The Harven.ai database layer has significant technical debt accumulated from rapid prototyping. The database was built incrementally with no migration files, no RLS policies, and no foreign key validation.

**UPDATE (2026-02-25):** Since the original audit, JWT authentication (HS256 with bcrypt) has been implemented with role-based access control. Password hashing now uses bcrypt with auto-migration from legacy plaintext. However, a plaintext fallback still exists in the login flow, and a hardcoded JWT secret fallback remains. The single-file backend (`main.py`, 4,779 lines, 81 routes) still performs all database operations through the Supabase Python client with no ORM, no query optimization layer, and pervasive N+1 query patterns.

### Severity Distribution

| Severity | Count | Description |
|----------|-------|-------------|
| CRITICAL | 6 | Must fix before any production deployment |
| HIGH | 8 | Significant risk, fix within next sprint |
| MEDIUM | 10 | Should fix, plan for upcoming sprints |
| LOW | 5 | Nice to have, backlog items |

## 2. Schema Issues

### 2.1 [HIGH → PARTIALLY RESOLVED] Password Storage

**File:** `backend/main.py`
**Table:** `users`

**UPDATE (2026-02-25):** Password hashing with bcrypt is now implemented. The login flow:
1. First attempts bcrypt verification via `pwd_context.verify()`
2. Falls back to legacy plaintext comparison if bcrypt fails
3. Auto-migrates legacy passwords to bcrypt hash on successful login

**Remaining issue:** Legacy plaintext fallback still exists. Users with old unhashed passwords are vulnerable to timing attacks until they log in and trigger auto-migration.

**Impact:** Reduced from CRITICAL to HIGH. Auto-migration helps but plaintext fallback remains a risk.
**Severity:** HIGH (was CRITICAL)

---

### 2.2 [HIGH] No Schema Migrations

No migration files exist anywhere in the repository. The schema was built directly in the Supabase dashboard. This means:

- No version control of schema changes
- No way to reproduce the database from scratch
- No rollback capability
- No way to verify schema drift between environments
- Column existence is uncertain (code contains trial-and-error column name attempts)

**Evidence:** Lines 1279, 1863-1868, 1933 -- the code tries multiple column names in sequence:
```python
for column_name in ["avatar_url", "avatar", "profile_image", "image_url"]:
    try:
        supabase.table("users").update({column_name: public_url})...
```

**Severity:** HIGH

## 3. Security Issues

### 3.1 [HIGH → PARTIALLY RESOLVED] Authentication / Authorization

**File:** `backend/main.py`

**UPDATE (2026-02-25):** JWT authentication (HS256) is now implemented:
- `HTTPBearer` security scheme
- `get_current_user()` / `require_auth()` / `require_role(*roles)` dependencies
- JWT tokens with configurable expiration (default 8 hours)
- Rate limiting on login (5/min), uploads (10/min), AI (10/min)

**Remaining issues:**
- JWT secret has a hardcoded fallback (`"harven-dev-secret-change-in-production"`)
- No token blacklist/revocation mechanism
- No refresh token flow
- Some admin endpoints may not enforce auth consistently
- Most endpoints lack rate limiting

**Impact:** Reduced from CRITICAL to HIGH. Auth exists but has gaps.
**Severity:** HIGH (was CRITICAL)

---

### 3.2 [CRITICAL] No RLS Policies

No Row Level Security policies exist. The backend uses the Supabase service key which bypasses RLS entirely. If anyone discovers the Supabase URL and anon key (which are exposed in the frontend code), they can read and write all data directly via the Supabase REST API.

**Evidence:** `harven.ai-platform-mockup/lib/supabase.ts` creates a Supabase client with the anon key, and there is no RLS to restrict what that client can access.

**Severity:** CRITICAL

---

### 3.3 [CRITICAL] API Keys Stored in Database

**Table:** `system_settings`

Sensitive credentials are stored in the `system_settings` table as plaintext:

| Column | Content |
|--------|---------|
| `openai_key` | OpenAI API key |
| `moodle_token` | Moodle API token |
| `jacad_api_key` | JACAD API key |
| `smtp_password` | SMTP password |
| `moodle_webhook_secret` | Webhook HMAC secret |
| `firewall_blocked_ips` | Security configuration |

These are accessible via `GET /admin/settings` which has no auth protection.

**Severity:** CRITICAL

---

### 3.4 [RESOLVED] Password Check

**UPDATE (2026-02-25):** Password verification is now functional:
- Primary: bcrypt via `pwd_context.verify()`
- Fallback: legacy plaintext comparison (raises 401 on failure)
- Auto-migration to bcrypt on successful legacy login

**Severity:** RESOLVED (legacy fallback tracked in 2.1)

## 4. Performance Issues

### 4.1 [HIGH] Pervasive N+1 Query Pattern

The most severe performance problem. Multiple endpoints execute queries in loops:

**`GET /disciplines` (lines 273-292):**
For each discipline, two additional queries run to count students and courses:
```python
for d in disciplines:
    students_count_res = supabase.table("discipline_students")...  # Query per discipline
    courses_count_res = supabase.table("courses")...                # Query per discipline
```
For 50 disciplines, this is 100+ extra queries.

**`GET /classes/{id}/courses` (lines 1604-1614):**
For each course, an additional query counts chapters:
```python
for course in courses:
    chapters_res = supabase.table("chapters")...eq("course_id", course['id'])...
```

**`GET /users/{id}/stats` (lines 3396-3434):**
At least 7 sequential queries per call: completed courses, started courses, completed IDs, total activities, last activity, time activities, quiz activities.

**`GET /users/{id}/achievements` (lines 3593-3740):**
Calls `get_user_stats()` (7 queries) + 2 additional queries for certificates and quizzes, totaling 9+ queries per call.

**`GET /users/{id}/courses/{course_id}/progress` (lines 3789-3836):**
4 sequential queries: course_progress, chapters, contents, completed activities.

**Severity:** HIGH

---

### 4.2 [HIGH] No Indexes Defined

No index definitions exist in the codebase. While Supabase auto-creates indexes on primary keys, the following commonly-filtered columns likely lack indexes:

- `users.ra` (used in every login)
- `users.role` (used in user listing)
- `discipline_teachers.teacher_id`
- `discipline_students.student_id`
- `courses.discipline_id`
- `chapters.course_id`
- `contents.chapter_id`
- `questions.content_id`
- `user_activities.user_id` + `action` (composite)
- `chat_sessions.user_id` + `content_id` (composite)
- `notifications.user_id` + `read` (composite)
- `system_logs.created_at`

**Impact:** Full table scans on every filtered query.
**Severity:** HIGH

---

### 4.3 [MEDIUM] No Pagination on List Endpoints

Most list endpoints return **all** records without pagination:

| Endpoint | Issue |
|----------|-------|
| `GET /disciplines` | Returns all disciplines |
| `GET /courses` | Returns all courses |
| `GET /users` | Returns all users |
| `GET /disciplines/{id}/teachers` | Returns all teachers |
| `GET /disciplines/{id}/students` | Returns all students |
| `GET /courses/{id}/chapters` | Returns all chapters + contents |

Only `GET /admin/logs/search` and `GET /users/{id}/activities` implement pagination.

**Impact:** As data grows, response times and payload sizes will increase linearly.
**Severity:** MEDIUM

---

### 4.4 [MEDIUM] Redundant Queries

Several endpoints query the same data multiple times:

- `get_user_stats()` queries `user_activities` with `action='course_completed'` twice (lines 3398 and 3403)
- `get_user_achievements()` calls `get_user_stats()` which already fetches the same activity data
- `get_dashboard_stats()` calls both `get_user_stats()` and `get_user_achievements()`, compounding the redundancy
- `get_admin_stats()` makes separate count queries for users and disciplines instead of a single aggregation

**Severity:** MEDIUM

---

### 4.5 [MEDIUM] Manual Joins Instead of PostgREST Relations

The code performs manual multi-step joins (fetch IDs, then fetch by IDs) instead of using Supabase's relation syntax:

```python
# Current pattern (2 queries):
response = supabase.table("discipline_teachers").select("teacher_id").eq("discipline_id", id)
teacher_ids = [row['teacher_id'] for row in response.data]
users_response = supabase.table("users").select("*").in_("id", teacher_ids)

# Could be (1 query):
response = supabase.table("discipline_teachers").select("*, users(*)").eq("discipline_id", id)
```

This pattern appears in: get_discipline_teachers, get_discipline_students, get_course_chapters (with contents), export_course.

**Severity:** MEDIUM

---

### 4.6 [LOW] Synchronous Performance Metrics

`GET /admin/performance` (line 773) calls `psutil.cpu_percent(interval=0.5)` which blocks the async event loop for 500ms. This should use `asyncio.to_thread()`.

**Severity:** LOW

## 5. Data Integrity Issues

### 5.1 [CRITICAL] No Foreign Key Enforcement

There is no evidence of foreign key constraints at the database level. The application code does not validate referential integrity before inserts or deletes:

- **Deleting a course** (line 1804) does not cascade to or check for dependent chapters, contents, or questions
- **Deleting a chapter** (line 2025) does not cascade to contents or questions
- **Deleting a content** (line 2068) does not cascade to questions or chat_sessions
- **Deleting a user** -- no endpoint exists, but if done manually, orphaned records would remain in discipline_teachers, discipline_students, notifications, user_activities, certificates, chat_sessions

The docstrings claim cascading behavior ("also removes associated contents") but the code only deletes from the target table.

**Severity:** CRITICAL

---

### 5.2 [HIGH] No Unique Constraints on Junction Tables

The `discipline_teachers` and `discipline_students` junction tables have no unique constraint on `(discipline_id, teacher_id)` or `(discipline_id, student_id)`. This allows duplicate enrollments:

```python
# Line 509-512: No duplicate check before insert
data = {
    "id": str(uuid.uuid4()),
    "discipline_id": discipline_id,
    "student_id": assignment.student_id
}
response = supabase.table("discipline_students").insert(data).execute()
```

The batch insert endpoint (line 540) acknowledges this problem in comments but does nothing about it.

**Severity:** HIGH

---

### 5.3 [HIGH] No Transaction Support

The Supabase Python client does not support transactions. Multi-step operations can leave data in inconsistent states:

- **Batch question update** (lines 2147-2168): Deletes all existing questions, then inserts new ones. If the insert fails, all questions are lost.
- **Course completion** (lines 3838-3900): Inserts activity, then checks progress, then potentially inserts another activity. Failure at any step leaves partial data.
- **Moodle export** (lines 4091-4224): Reads session, messages, user, content, course, chapter (6 queries), formats data, then updates session status. Any failure mid-process can leave an inconsistent export state.

**Severity:** HIGH

---

### 5.4 [MEDIUM] Orphan Data Risk on Delete

Course deletion (line 1804) only deletes the course row:
```python
response = supabase.table("courses").delete().eq("id", course_id).execute()
```

This leaves orphaned records in: `chapters`, `contents`, `questions`, `chat_sessions`, `chat_messages`, `user_activities` (target_id references), `certificates`, `course_progress`.

Similarly, chapter deletion (line 2025) orphans: `contents`, `questions`.

**Severity:** MEDIUM

---

### 5.5 [MEDIUM] Race Conditions in Chat Session Counter

Lines 4044-4047 implement a read-then-write counter update without locking:

```python
session = supabase.table("chat_sessions").select("total_messages").eq("id", session_id).single().execute()
current_count = session.data.get('total_messages', 0) if session.data else 0
supabase.table("chat_sessions").update({"total_messages": current_count + 1}).eq("id", session_id).execute()
```

Concurrent message sends can lose count increments. Should use PostgreSQL's atomic increment: `total_messages = total_messages + 1`.

**Severity:** MEDIUM

---

### 5.6 [MEDIUM] `"now()"` String Instead of Server Timestamp

Throughout the code, `created_at` is set to the string `"now()"`:
```python
"created_at": "now()"
```

This relies on PostgREST interpreting `"now()"` as the PostgreSQL function. If the column type is TEXT instead of TIMESTAMPTZ, this will store the literal string `"now()"`. This behavior is fragile and database-implementation-dependent.

**Severity:** MEDIUM

---

### 5.7 [LOW] No Soft Deletes

All delete operations are hard deletes. There is no `deleted_at` column or soft delete pattern. Accidental deletions are unrecoverable.

**Severity:** LOW

---

### 5.8 [LOW] No `updated_at` Timestamps

Most tables lack an `updated_at` column. Only `system_settings` and `chat_sessions` reference `updated_at`. This makes it impossible to track when records were last modified.

**Severity:** LOW

## 7. API-DB Interaction Patterns

### 7.1 Architecture Overview

```
Frontend (React)
    |
    | HTTP (fetch to localhost:8000)
    v
FastAPI Backend (main.py - single file, 4577 lines)
    |
    | supabase-py client (PostgREST)
    v
Supabase (PostgreSQL)
```

- **No ORM:** All queries use the Supabase Python client (`supabase.table().select().eq().execute()`)
- **No query abstraction layer:** Every endpoint contains inline database queries
- **No connection pooling configuration:** Single global Supabase client
- **No caching:** Every request hits the database

### 7.2 Query Patterns Used

| Pattern | Frequency | Example |
|---------|-----------|---------|
| `select("*")` | Very common | Almost all read endpoints |
| `select("id", count="exact")` | Common | Count queries |
| `eq()` filter | Very common | Single-value equality filter |
| `in_()` filter | Common | Multi-value filter (after manual join) |
| `ilike()` | Rare | Search endpoint only |
| `or_()` | Rare | Log search only |
| `order()` | Common | Sorting results |
| `limit()` | Uncommon | Only a few endpoints |
| `range()` | Rare | Only paginated endpoints |
| `single()` | Common | Single-record lookups |
| `.insert()` | Common | Creates |
| `.update()` | Common | Updates |
| `.delete()` | Common | Hard deletes |
| `.match()` | Rare | Composite key delete (line 468) |

### 7.3 Supabase Client Configuration

```python
supabase: Client = None  # Global mutable state

@asynccontextmanager
async def lifespan(app: FastAPI):
    global supabase
    supabase = create_client(SUPABASE_URL, SUPABASE_KEY)
    yield
```

- Uses a single global client instance
- No connection pooling settings
- No retry logic
- No query timeout configuration
- Client initialization failure does not prevent app startup (prints warning, continues with `supabase = None`)

### 7.4 Error Handling Pattern

All endpoints follow the same pattern:
```python
try:
    # Supabase query
except Exception as e:
    print(f"Error: {e}")
    return []  # or raise HTTPException
```

This catches all exceptions indiscriminately, including network errors, schema errors, and data errors. Many endpoints return empty arrays on error instead of proper HTTP error codes.

### 7.5 Notable Anti-Patterns

1. **Trial-and-error column updates:** The upload endpoints try multiple column names until one works (lines 1279-1288, 1863-1879, 1933-1942)
2. **Swallowed errors:** Many catch blocks use `pass` or return empty data
3. **No request validation against DB schema:** Pydantic models don't match DB columns
4. **Settings table as config store:** Runtime configuration including API keys stored in DB
5. **`select("*")` everywhere:** No column selection optimization; always fetches all columns
